1. Overall User Testing Summary

This user testing summary generated by UsableNet AQA in conjunction with accessibility reviewers and user testers, outlines the qualitative evaluation of accessibility based on real user feedback.

1.1. Methodology

The following report captures the results of user testing performed by a selected set of users. The user testing data is captured and organized within UsableNet AQA to provide reference and detail of each user experience along with views of that data in a summary format for different stakeholders. Below the sections highlight key areas of the report with context on how it is complied.

Task Completion

User testers are asked to complete a number of tasks following key user journeys. The first item summaries the data collected and presented in "Task Completion Success" table. The users indicate if the task was completed or not, they can also indicate how long the task took. The Summary chart of task completion focuses on the success while later in the "User Testing Data" you can review how long it took each tester to complete the task.

Recommendation Summary

Based on the overall observations by each user, a set of recommendations is established. These recomendations are given a ranking from Low to High and the resulting Recommendation Summary table allows the most important observation to be reviewed and action taken.

User Testing Rating

Whenever users complete or not a task, they are asked to rate the user journey against a given set of guidelines in from of a questionnaire to establish the overall experience. The user is asked to rate the experience for each guideline from 1 to 5, with 5 indicating a very good experience and 1 indicating a very poor experience. UsableNet AQA then calculates the rating for each user journey based on all user testers scores along with an overall rating of the experience across all user journeys tested.

User Testing Data

All the user data collected within UsableNet AQA is organized per user journey. The overview of each journey provides the detail needed to establish which one needs the most attention. The detailed information includes user comments on questionnaire and feedback on the challenges and barriers presented to the testers. Users can also add further observations about the tested journey if they feel required.

    Instructions

    1. You and your 1 and 10-year-old daughters decided to take a tour of New York where your wife works as a soldier, so you can be together for the weekend, and then travel to vacation by plane. You hop on the Amtrak site to buy one way tickets for you and the kids, adding your passenger information and assigning the seats in the process. After filling in all the necessary information, you went to the payment page and booked tickets for all three of you. Upon landing on the Confirmation page (where you saw the confirmation of your reservation), you decided to change the seats.

      • Please note: You can use the fake Credit card for testing purposes.

    2. Task can be considered as complete when you choose to travel in one way from Washington to New York, add two kids as military children (one child and one infant), reserve a seat for everyone and after buying a ticket, change seats for all three.

1.4. High priority recommendations

High priority recommendations table
User Journey Recommendation
Task 1

UsableNet User Tester 3 quote:

  • There was no heading for the booking content section on the homepage.

  • There were several unlabeled graphics on the homepage.

  • The "One Way" button was not labeled as a menu.

  • The departure/arrival station fields were not accessible using arrow navigation.

  • The fields for the departure/arrival stations were not initially labeled as edit fields e.g. departure station was read by JAWS as a check box.

  • When populating the station fields, a list of suggested stations was sometimes available. However, it was not consistent.

  • The button to add passengers was located in the tab order after the "Find Tickets" button.

  • The buttons to increment/decrement number of passengers for each passenger category were unlabeled.

  • When a button for a fare was selected, the button did not indicate that it was selected.

  • There was no heading for the cart content.

  • There was no clear button to proceed to the checkout from the cart, and the button that was required to continue was labeled as "Continue to Next Page" , which was located at the bottom of the page below the list of trains available, and could be confused as part of the pagination links which were located directly above those buttons.

  • There were unlabeled buttons on the passenger information pages.

  • After selecting the insurance focus was moved to the payment information section of the page. However, this bypassed a check box to accept the terms and conditions.

  • After populating all the payment fields, the "Purchase Trip" button was disabled. After navigating through the page, and selecting the terms and conditions check box the button was still disabled. An attempt to select the button was made, and the transaction could not be finalized.

  • There was no error reporting to indicate, if there was an error within the form.

**These comments were added once assistance was provided to work around the issue to complete the transaction. **

  • It was not clear that the "Add" button had to be used in order to complete the transaction.

  • On the confirmation page, when the "Trip Tools" button was selected, JAWS indicated that the space bar should be entered followed by the use of arrow keys. When these actions were perform focus would not move.

  • When the trip Tools button was expanded all choices in the menu were read twice, sometimes they both read as buttons, and other times, one of the choices would be read as a link. If the button was selected the trip tools button would be collapsed.

  • If the "Modify Trip" link was selected, there was no information on the page to select a seat.

This is a high priority issue.
